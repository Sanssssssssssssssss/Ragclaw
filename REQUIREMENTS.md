# REQUIREMENTS

## 需求状态说明
- `Confirmed`：已确认需求
- `Assumption`：当前采用的假设，后续可修订
- `Open`：缺失信息，待确认

## 功能需求

### FR-001 项目记忆机制
- 状态：Confirmed
- 系统必须使用以下文件作为长期记忆：
  - `PROJECT_BRIEF.md`
  - `REQUIREMENTS.md`
  - `ARCHITECTURE.md`
  - `DECISIONS.md`
  - `TASKS.md`
  - `STATE.md`
- 验收标准：
  - 上述文件在仓库内存在。
  - 每轮任务都可以基于这些文件重建上下文。

### FR-002 私有仓库作为主线存储
- 状态：Confirmed
- 项目必须以我们的私有 GitHub 仓库作为主线开发与更新存储。
- 验收标准：
  - 本地仓库能关联到私有远程仓库。
  - 后续变更默认提交到私有仓库。

### FR-003 参考并复现上游基线
- 状态：Confirmed
- 项目第一阶段必须参考 `lyxhnu/Skill-First-Hybrid-RAG`，实现与其接近的最小可运行能力。
- 基线目标至少包括：
  - 后端 API 可启动
  - 前端界面可启动
  - 基础聊天链路可用
  - 本地知识检索链路可用
- 验收标准：
  - 可以本地完成前后端启动。
  - 可以执行至少一次端到端访问验证。

### FR-004 Skill-first 混合检索学习主线
- 状态：Confirmed
- 项目必须保留“Skill-first，检索兜底”的学习主线，避免初始化阶段偏离参考项目核心思路。
- 验收标准：
  - 架构文档明确记录该检索主线。
  - 第一版实现中不主动移除该机制。

### FR-005 后续研究扩展能力
- 状态：Confirmed
- 项目需要支持后续扩展以下研究方向：
  - 多模态 RAG
  - GraphRAG / RAGGraph
  - 评测与实验管理
- 验收标准：
  - 架构上预留扩展模块或实验区。
  - 任务列表中包含相应规划项。

### FR-006 持续化状态管理
- 状态：Confirmed
- 每轮任务完成后，需要记录当前状态、下一步、风险和接班说明。
- 验收标准：
  - `STATE.md` 持续维护。
  - `TASKS.md` 至少包含待办、进行中、已完成、阻塞四个区段。

## 非功能需求

### NFR-001 可维护性
- 状态：Confirmed
- 优先选择简单、稳健、可维护的方案，避免为未来假设场景过度设计。

### NFR-002 可审计性
- 状态：Confirmed
- 记忆、技能、知识材料、配置和关键决策应尽可能以可见文件形式存在。

### NFR-003 本地优先
- 状态：Confirmed
- 第一阶段应优先支持本地开发运行，不依赖复杂外部基础设施。

### NFR-004 Windows 兼容性
- 状态：Assumption
- 当前优先保证在 Windows 本地环境可执行。

### NFR-005 演进安全性
- 状态：Confirmed
- 新方向研究不应轻易破坏已验证的基线能力。

## 业务规则
- BR-001：未映射到本文件的实现，不默认进入正式范围。
- BR-002：若聊天内容与项目文件冲突，以最新确认的项目文件为准，并明确标注冲突。
- BR-003：需求不完整时，必须记录缺失信息与采用的假设。
- BR-004：在未完成基线验证前，不进行大规模架构推翻。

## 里程碑验收标准

### M1 初始化完成
- 项目记忆文件齐备。
- 任务、状态、决策机制建立完成。

### M2 基线代码入库
- 上游参考代码进入我们的主仓开发流程。
- 本地依赖安装路径和启动步骤明确。

### M3 第一个可运行版本
- 后端健康检查通过。
- 前端可访问。
- 基础聊天与至少一条检索链路验证通过。

### M4 扩展实验准备完成
- 形成清晰的实验分层方案。
- 至少定义一个扩展方向的实现入口。
